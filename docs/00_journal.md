| 2026-05-19 | ES housekeeping fields belong in ingest pipelines, not Python writers | A request to add `create_date` to `chunks_v1` triggered the reflex to plumb the timestamp through every Python write path — `DocumentEmbedder` (current real writer), `VectorExtractor` (legacy stub but still alive), the dual-write paths under `EmbeddingLifecycleService`, and any future writer would each have to remember to stamp `created_at=utcnow().isoformat()` into the chunk `_source`. That design distributes time-source responsibility across N writers, opens application clock-skew between worker pods, forces every retry / supersede / cutover code path to re-honour the convention, and makes the timestamp invisible to anyone who only reads the index mapping. The ES-native primitive (an **ingest pipeline** with a `set` processor on `{{{_ingest.timestamp}}}`, referenced from index settings via `index.default_pipeline`) achieves the same result in the data plane with **zero Python writer awareness** and a single authoritative clock (the ES coordinating node). The same reasoning generalises to any write-time housekeeping field: simple derived values (hash of `text` for dedupe), enrichments (geo-IP, lang detect via ML inference processor), or default-stamping (`schema_version`). Industry references: Elastic's own guidance for `@timestamp` on time-series indices, OpenSearch ISM lifecycle pipelines, and the Logstash → Beats → ES topology — all push housekeeping out of producer code and into pipeline processors. | First-look design defaulted to "where in the writer code do I add the field?" without asking "does the storage engine already have a primitive for this?" Compounding this, the project's `DuplicatePolicy.OVERWRITE` write convention creates a subtle trap: `set { override: false }` on an ingest pipeline checks only the **incoming `_source`** for the field's presence, **not** the existing doc in the index. Under OVERWRITE every retry is an `index` op that fully replaces `_source`, so `override:false` does NOT preserve a "first-write" timestamp — preserving first-write would require a `script` processor that does an in-place GET of the previous doc, which is slow, racey, and breaks bulk throughput.
