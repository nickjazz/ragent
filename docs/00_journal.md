| 2026-05-04 | Retrieval/ACL | Earlier round 3 declared "OpenFGA out-of-scope across all phases" and proposed an owner-based ES filter (`terms(owner_user_id ∈ [user_id])`) as the P2 ACL. When reversing a prior Decision-Log row, append a new dated row that explicitly cites the superseded ID — never edit the old row in place. |
| 2026-05-04 | Interface freeze | B15 added "embed `f"{title}\n\n{chunk_text}"`" to `VectorExtractor.extract` after the Plugin Protocol was frozen — but the Protocol signature is `extract(document_id) -> None` with no way to pass `source_title`. | 2026-05-04 | Capacity planning | B2 capped CSV at 50 MB ("no row cap, inherit file-size limit") with no worst-case math. Round-4 audit ran the multiplication: 50 MB × 50 chars/row × 1 sentence/row × 30 s/embed ≈ 250 hours. The "no cap" decision was implicitly assuming small CSVs; large CSVs broke the system's wall-clock budget silently. B24 (RowMerger) restored the bound to ~25 k chunks. | "No cap" decisions are easy to make and hard to verify. The cost surfaces as a worst-case scenario the spec author didn't compute. | **[Rule]** Every spec decision of the form "no cap on N" / "bounded only by X" **MUST include the worst-case multiplication** in the same paragraph: `worst_case = max_X × cost_per_unit`. If the result exceeds an existing system budget (timeout, memory, count, cost), the "no cap" is implicit and false — introduce packing, aggregation, or a hard cap before the decision is accepted. Applies to file-size limits, list endpoints, fan-out plugins, retry counts. |
| 2026-05-04 | DB design | `documents.storage_uri` stored full URIs like `minio://ragent-staging/<key>` even though the bucket name is constant per deployment. Renaming the bucket would require an `UPDATE documents SET storage_uri=...` migration. B25 split the data into `object_key` (per-row) + `MINIO_BUCKET` (env-driven config). | Storing immutable per-deployment values inside per-row data conflates two lifecycles (config rotation vs row mutation). Defaulted because "URI is one string" felt simpler than two columns of split data. | **[Rule]** Columns storing values that are **constant for the deployment** (bucket name, region, hostname, default queue, cluster id) belong in **config**, not in rows. Litmus test: "if I rename this constant tomorrow, do I need to `UPDATE` every existing row?" If yes, the value is misplaced. Apply at schema-design time; once a row contains it, the migration cost compounds with the table size. |
| 2026-05-04 | ES Field Classes | Adding chat filter fields (`source_app`, `source_workspace`) re-opened the "should this go on the ES index?" debate that B14 (`owner_user_id` reversal) had previously settled with a "no". Risk: applying B14 too broadly would have forced an over-fetch+JOIN post-retrieval filter, silently truncating narrow scopes. Reality: B14 was specifically about **authorization** fields; **scope/content metadata** that already drives row identity (like `lang`, `title`) belongs on the index. | The lesson from B14 was over-generalised in conversational shorthand to "no business fields on ES". The actual rule is narrower — auth fields specifically. Without a written distinction, the next person making the call has to re-derive it under deadline pressure. | **[Rule]** Distinguish two field classes when deciding whether to denormalise onto a retrieval index: **(1) Authorization fields** (answer "can this caller see this row?" — owner, viewer set, ACL relation): NEVER on the retrieval index, regardless of phase. Use a post-retrieval Permission Layer Protocol. **(2) Scope/content metadata** (answer "what kind of content is this?" — language, source app, workspace, content type, title): denormalise freely when the access pattern requires filtering at scoring time. Litmus test: "if this field's value changes for a row, is it still fundamentally the same record?" Yes ⇒ scope metadata, denormalise. No (the row's identity changes) ⇒ likely auth-adjacent, keep off the index. | 2026-05-04 | Typed dispatch | Initial `_fan_out_plugins` used `method: str` + `getattr(p, method)` to switch between `extract` and `delete`. This is a stringly-typed two-branch dispatch: a typo compiles fine and surfaces only at runtime as `AttributeError`; static analysis and IDEs cannot trace the call. Caught in `/simplify` review, not in tests. | "Pass a string and getattr it" felt shorter than "pass a callable" when drafting the initial implementation. The Protocol was already imported for type hints but not used to define the callable type. | **[Rule]** Two-branch method selection (`"extract"` vs `"delete"`, `"read"` vs `"write"`) MUST use typed `Callable` parameters, not `str` + `getattr`. The caller already holds the plugin reference; extract the callable at the call site (`p.extract`, `p.delete`) and pass it down. `getattr(obj, str)` is acceptable only for user-supplied dynamic attribute names (e.g. config-driven field names); never for internal dispatch between a known-finite set of methods. |
| 2026-05-04 | Bootstrap | `init_schema.py` split schema.sql by `;` and checked `if stmt.startswith("--")` to skip comments. The first statement chunk included the header comment block AND `CREATE TABLE documents` — starting with `--`, so the whole chunk was filtered. Only the second table (chunks) was created. Silent data-only write caused document-table insertions to fail at the first integration run. | SQL files often open with multi-line comment headers that precede the first DDL statement. Treating the comment prefix as a proxy for "skip this statement" ignores that real statements can follow on subsequent lines within the same split fragment. | **[Rule]** SQL statement extraction from `.sql` files MUST strip comment lines line-by-line before checking if the remaining fragment is non-empty — never apply `startswith("--")` to the whole fragment. Use a helper like `_strip_comments(raw)` that filters each line independently. Apply the same pattern to any code that parses SQL files (migration runners, schema-snapshot appliers, test fixtures). |
| 2026-05-04 | FastAPI | FastAPI parameters declared as bare default values using `File(...)`, `Form(...)`, `Header(...)` trigger ruff B008 ("Do not perform function call in default argument"). The failure appeared only at lint time, blocking CI while tests had already passed. Switching to `Annotated[Type, Dependency()]` = None resolved both the lint violation and improves FastAPI's OpenAPI schema generation. | FastAPI's older documented style used bare defaults; ruff B008 was added to catch exactly this class of bug (default evaluated once at import, not per call). The team copied the older style without checking ruff's rules. | **[Rule]** All FastAPI parameter declarations MUST use the `Annotated[Type, Dependency()]` pattern (not bare default `= Dependency()`). This applies to `File`, `Form`, `Header`, `Query`, `Body`, `Depends`, and any FastAPI-injected type. | 2026-05-04 | SQLAlchemy | `composition.py` used `with engine.connect() as conn:` and passed `conn` to repository constructors. `with` closes the connection at block exit, leaving repositories holding a closed connection that raises `InvalidRequestError` on the first query. The naive "fix" — hoist out of the `with` and hold one app-lifetime `Connection` — would have violated `00_rule.md` §Database Practices (Mandatory Connection Pool) and reintroduced the "Packet sequence error" / "command out of sync" failure mode under concurrent FastAPI traffic. | Two compounding mistakes: (1) the `with` context manager was applied to a resource passed into a longer-lived owner; (2) the dependency direction was wrong — singletons should not own a `Connection` at all. SQLAlchemy's context manager silently invalidates the connection on exit rather than raising at assignment time, masking the lifetime mismatch. | **[Rule]** Singletons (repositories, services, the composition root) MUST hold the `Engine`, **never** a `Connection`. Connections are checked out per unit of work via `with engine.begin() as conn:` (write) or `with engine.connect() as conn:` (read) inside each repository method, or injected per-request through a `Depends`-driven session factory. This satisfies `00_rule.md` §Database Practices and the present concurrency-safety requirement in one move. Forbidden patterns: `self._conn = engine.connect()` at `__init__`; passing a `Connection` (rather than an `Engine`) into a constructor; any module-scope `connection = engine.connect()`. Use `with engine.connect() as probe: probe.execute(text("SELECT 1"))` for one-shot startup health checks only. |
| 2026-05-04 | DI | `app.py` initially passed `broker=None` to `IngestService`. `IngestService.create()` unconditionally calls `self._broker.enqueue(...)`, so the first `/ingest` POST would have raised `AttributeError: 'NoneType' object has no attribute 'enqueue'`. Without an integration test covering the real wiring, this class of error is invisible to the unit test suite. | **[Rule]** Never pass `None` for a required DI dependency unless the callee explicitly handles `None` (e.g. optional feature). When a constructor argument is typed non-Optional, pass the real instance. If the real instance requires env vars not available in test, use a mock in tests — do not default the production path to `None`. Review every `SomeClass(dep=None)` call site before committing; if the callee calls `dep.method()` unconditionally, `None` is a latent `AttributeError`. |
| 2026-05-04 | Container design | Initial `Container` exposed `_chat_router_kwargs: dict` (a private dict embedding `rate_limit` and `rate_limit_window`) accessed via `**container._chat_router_kwargs` in `app.py`. | 2026-05-09 | Observability contract | Audit of the ingest→search→chat flow surfaced two coupled gaps: (a) **distributed seams emit no logs** — `ingest.dispatched` (producer-side after `kiq`), `ingest.task.started` (worker-side pickup), `ingest.deleted` (soft-delete service), `supersede.completed` (task tail), and the chat hydrator's `status='READY'` filter (silent drop) all had no business event, so an operator could not grep "did the message land?", "did the worker pick it up?", or "how many chunks were silently filtered out by the READY gate?"; (b) **the global `Exception` handler collapsed every typed domain exception into `error_code="INTERNAL_ERROR"`** — `IngestStepError(error_code="EMBEDDER_ERROR")` and any future upstream-service exception lost their domain code on the way out; the response body said "internal error" while the only place that knew it was an embedder failure was a worker log nobody was reading. | 2026-05-16 | Feedback as ES index | T-FB.* introduced a non-parametric supervised-learning signal — user feedback (`like_count`/`dislike_count` per source_id) — to RAG retrieval without building a training pipeline or storing chat content in MariaDB. (5) Source-level (not chunk-level) feedback survives re-ingest because `documents.(source_app, source_id)` is the stable external identity; chunk_id changes on every re-split. ## SRE

| Date | Topic | Description | Root Cause | Actionable Guideline |
| :--- | :--- | :--- | :--- | :--- |
| 2026-05-04 | Concurrency | Concurrent worker + Reconciler can both update `documents.status`/`attempt`, causing double-dispatch. | No explicit locking on state-machine mutations. | **[Rule]** All `documents.status` mutations use `SELECT … FOR UPDATE` (Worker) or `FOR UPDATE SKIP LOCKED` (Reconciler). Repository.`update_status` raises `IllegalStateTransition`. |
| 2026-05-04 | Cleanup Ordering | Worker deleted MinIO staging object **before** committing terminal `status=READY`, so a crash in the gap caused the next reconciler retry to 404 on MinIO and eventually mark the document `FAILED` despite a successful pipeline run. Mirror risk: supersede ordered by `created_at < self` was not order-safe — a newer doc finishing READY before an older doc left the older one orphaned. | Side-effect cleanup (and stateful comparisons) ordered before / against durable state commits. | **[Rule]** Within any worker, **durable state commits MUST precede external side-effect cleanup**: (1) commit terminal status; (2) best-effort cleanup of staging artefacts; (3) log orphan-tolerated event. For convergence rules across multiple writers (e.g. supersede), use a **set-based survivor selection** (`MAX`/`MIN` over the candidate set under `FOR UPDATE SKIP LOCKED`) rather than a self-referential pairwise comparison. Applies to MinIO, Redis staging, and any future transient store. |
| 2026-05-04 | Lock Scope | Long-running pipeline held `SELECT … FOR UPDATE` on the document row across the entire 60+ s pipeline body (embedder, ES, plugins). A live-but-hung worker would never release the lock, and the Reconciler's `SKIP LOCKED` would skip the row forever — silent stuck-PENDING. Worker process *death* releases the lock (DB drops session); worker *hang* does not. | Single transaction wrapped both the cheap status mutation and the expensive external pipeline body. | **[Rule]** Status mutations and external work MUST use **separate short transactions**.
